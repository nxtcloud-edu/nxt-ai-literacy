# 작업 명령서 (Work Orders) — 운영 규칙 (템플릿)

## 파일 규칙
- `WO-NNN-<슬러그>.md`, 상태 머신: `대기 → 진행 중(<Coder>) → 검증 대기 → 완료/반려(사유)`
- 작성은 Planner, 실행은 Coder, 판정은 Planner(검증자)

## 명령서 필수 섹션
목표 / **설계 결정 (변경 금지)** / 컨텍스트 (필독 파일) / 작업 단계 /
**완료 기준 (검증 가능한 조건)** / 검증 명령 / **금지 사항**

## 공통 제약 (모든 명령서에 적용)

### 절대 금지 — 위반 시 해당 WO 자동 반려
1. **클라우드·프로덕션 변경 일체 금지** (DB push, 배포 트리거, 프로덕션 접속, 클라우드 자격 파일
   읽기/source). 로컬 적용까지만. 프로덕션 반영은 **검증 통과 후 검증자만** — 분업의 핵심 안전장치.

### 워크트리·브랜치 게이트
1. Coder 작업 위치: `../<repo>-coder` 워크트리 (WO 시작 시 검증자가 `wo/NNN` 브랜치 준비)
2. **`wo/NNN`에만 커밋** — main 직접 커밋·푸시 금지
3. 머지는 검증자 전담: 검증 통과 → 팁==검증SHA 대조 → main 머지 → 배포
4. append-only 저널은 `.gitattributes`의 merge=union으로 충돌 방지
5. 전 스위트는 한 번에 하나만 (로컬 DB·포트 공유)
6. 완료 신호 = "커밋 + TURN_LOG 기록" — 상태 줄만 먼저 바꾸지 않는다

### 기타
- 자기 보고 불신 원칙: Commands 전수 기재, "미실행" 주장은 실측 대조됨
- 검증 없이 완료 기록 금지 / 커밋은 한 커밋 = 한 목적
- 백그라운드 프로세스는 턴 종료 전 정리

### 검증 범위 규칙 (WO-023 환류)
- 명령서의 브라우저 실측 요구는 **핵심 시나리오 최대 5개로 한정**해 명시한다.
  범위 없는 "충분히 확인" 지시는 코더의 검증 과잉 루프를 유발한다 (실측: WO-023에서
  코드 커밋 후 19분 검증 반복 → 플래너 인터럽트로 종료).
